# Проект "Шашки": многопользовательская онлайн-игра

Этот проект представляет собой реализацию классической игры в шашки с возможностью игры по сети и общением с соперником по чату, email-уведомления при подключении соперника. 

## Используемые технологии

* **Python**, **Flask** — веб-сервер и REST API
* **Flask-SocketIO** — real-time (WebSocket / long-polling)
* **smtplib** — отправка email через SMTP
* **HTML / CSS / JavaScript** — клиент
* **Railway** — деплой

## Об игре

Классические русские шашки для двух игроков по сети. Регистрация опциональна: можно играть как гость. Зарегистрированным пользователям начисляются кубки за победы.

## Email-уведомления (для защиты)

### Назначение

Когда второй игрок подключается к очереди, **ожидавший зарегистрированный** пользователь получает письмо на email, указанный при регистрации. Так игрок узнаёт о сопернике, даже если вкладка с игрой не активна.

Тема письма: **«Шашки — противник найден»**. В тексте — имя соперника и напоминание подтвердить готовность в игре.

### Кому отправляется

| Условие | Письмо |
|---------|--------|
| Игрок **зарегистрирован** и ждал в очереди | Да, на email из аккаунта |
| Игрок зашёл как **гость** | Нет |
| Второй игрок подключился (тот, кто пришёл позже) | Нет — письмо только ожидавшему |

### Как это работает (цепочка событий)

1. Пользователь регистрируется → email сохраняется в `auth.py` (в памяти сервера).
2. Игрок заходит в `/game` → WebSocket-событие `join_queue`.
3. Если соперник уже ждёт → создаётся партия, обоим приходит `match_found`.
4. Сервер в фоне вызывает `notify_waiting_player()` → `mailer.send_opponent_joined_email()`.
5. Письмо уходит через **Gmail SMTP** (`smtp.gmail.com:587`, STARTTLS).
6. Оба игрока нажимают **«Готов»** → только после этого начинается партия.

### Модули в коде

| Файл | Роль |
|------|------|
| `auth.py` | Хранение email при регистрации, `get_email(username)` |
| `mailer.py` | Формирование и отправка письма через `smtplib` |
| `socket_handlers.py` | Вызов уведомления при событии `join_queue` |
| `config.py` | Чтение SMTP-настроек из переменных окружения |

Отправка выполняется в **фоновой задаче** SocketIO (`start_background_task`), чтобы не блокировать подключение игроков.

### Безопасность

* Пароли пользователей хэшируются (`werkzeug.security`), в открытом виде не хранятся.
* Учётные данные SMTP хранятся в **переменных окружения** (`.env` локально, **Variables** на Railway), не в коде и не в git.
* Для Gmail используется **пароль приложения** (при включённой 2FA), а не основной пароль аккаунта.
* Один служебный Gmail-ящик рассылает письма всем игрокам; каждому игроку настраивать SMTP не нужно.

### Режим без SMTP

Если `SMTP_ENABLED=false`, письмо **не отправляется**, а текст выводится в лог сервера (`[EMAIL] To: ...`). Удобно для локальной отладки без почты.

### 2. Переменные окружения

| Переменная | Значение |
|------------|----------|
| `SMTP_ENABLED` | `true` |
| `SMTP_HOST` | `smtp.gmail.com` |
| `SMTP_PORT` | `587` |
| `SMTP_USER` | Gmail, с которого отправляются письма |
| `SMTP_PASSWORD` | пароль приложения Google |
| `MAIL_FROM` | тот же Gmail, что `SMTP_USER` |

### Локально

```bash
pip install -r requirements.txt
cp .env.example .env
# впишите SMTP_USER, SMTP_PASSWORD, MAIL_FROM
python3 app.py
```

Файл `.env` подхватывается через `python-dotenv`. В `.gitignore` он исключён.

### На Railway

**Service → Variables** — те же 6 переменных. Без кавычек: `true`, не `"true"`.

После деплоя в **логах** при старте должно быть:
```
[CONFIG] SMTP enabled=True, host=smtp.gmail.com, user=set
```
Если `enabled=False` или `user=MISSING` — переменные не подхватились (не тот сервис, опечатка в имени, нужен Redeploy).

Если в логе `SMTP выключен — письмо только в лог` — на Railway не задан `SMTP_ENABLED=true` или пустые `SMTP_USER` / `SMTP_PASSWORD`.

SMTP включается автоматически, если заданы и логин, и пароль (даже без `SMTP_ENABLED`).
